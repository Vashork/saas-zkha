## 1. Дашборд: статистика показывает 0 ₽ / 0 платежей

**Симптом:** Карточки «Ожидает оплаты» и «Просрочено» показывают 0, хотя подрядчики заведены и есть просроченные платежи.

**Вероятная причина:** Логика классификации в `dashboard.py` (строки 82-93) фильтрует платежи **только за выбранный месяц** (`Payment.year == year, Payment.month == month`). Если платежей за текущий/выбранный месяц нет — статистика пустая. Но просроченные платежи могут быть за другие месяцы.

**Что проверить:**
- Есть ли платежи в БД? `SELECT * FROM payments LIMIT 10;`
- Какие у них `year`, `month`, `status`?
- `_effective_status()` правильно определяет overdue?

**Решение (вариант А):** Карточки «Ожидает оплаты» / «Просрочено» должны показывать **глобальную** статистику по ВСЕМ неоплаченным платежам, а не только за выбранный месяц.

**Решение (вариант Б):** Карточки привязаны к выбранному месяцу, но если месяц пустой — показывать ближайший месяц с платежами.

> Рекомендую вариант А: pending/overdue = глобально, total/paid = за выбранный месяц.

**Файл:** `app/web/routes/dashboard.py` (строки 75-94)

## 3. Access Control: права не применяются для обычных пользователей

**Симптом:** Админ убирает у пользователя право на просмотр дашборда (снимает чекбокс), но пользователь всё равно видит дашборд.

**Вероятная причина:** `_require_page()` в `auth.py` читает `page_permissions` из **cookie**, но при сохранении прав в `/settings/users/{user_id}/update` cookie не обновляется у клиента. Cookie `page_permissions` устанавливается **при логине** и не обновляется при изменении прав админом.

**Текущая логика:**
1. При логине → сервер читает `user.page_permissions` из БД → ставит cookie `page_permissions`
2. Админ меняет права пользователя → БД обновляется, но cookie у пользователя НЕ меняется
3. Пользователь продолжает работать со старыми правами в cookie

**Решения:**
- **Вариант А (простое):** Убрать чтение из cookie, читать права из БД при каждом запросе (требует session/token для идентификации пользователя)
- **Вариант Б (быстрое):** При изменении прав пользователя — инвалидировать все его сессии (форсировать перелогин). Или: при `update_user` — вернуть redirect с заголовком `Set-Cookie` для обновления прав (но это cookie текущего пользователя-админа, а не целевого).
- **Вариант В (правильное):** Перейти на server-side sessions (не cookie permissions). Хранить `user_id` в signed cookie, а права читать из БД.

> Для быстрой фиксации: добавить предупреждение в UI «Пользователь должен перезайти, чтобы права вступили в силу». Для production — вариант В.

**Файлы:** `app/web/routes/auth.py` (`_require_page`, `login`, `update_user`)
