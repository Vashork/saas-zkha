## 2. Что изменено в последнем пакете

### Dashboard: счётчики открытых платежей без суммы

Пользователь заметил, что в карточках:

```text
Всего к оплате
4 500 ₽
1 открытых платеж(ей), включая без суммы
```

нужно учитывать платежи без суммы в количестве. Исправлено:

```text
Всего к оплате
4 500 ₽
2 открытых платеж(ей), включая 1 без суммы
```

и аналогично для просроченных:

```text
Из них просрочено
4 500 ₽
2 просроченных платеж(ей), включая 1 без суммы
```

Важно: рублёвая сумма не увеличивается по variable-платежу без начисления, потому что сумма ещё неизвестна. Увеличивается только количество открытых/просроченных строк.

Файлы:

- `app/web/routes/dashboard.py`
- `app/web/templates/dashboard.html`

## 4. Администрирование прикреплённых файлов

Пользователь хочет в настройках видеть все прикреплённые файлы и уметь удалять их с диска.

### Текущее устройство файлов

Файлы лежат в volume `./data:/app/data`, upload dir `/app/data/uploads`.

Обычно путь в БД хранится в `Payment.receipt_file`, например:

```text
2026/06/<uuid>.pdf
```

Физически файл лежит:

```text
./data/uploads/2026/06/<uuid>.pdf
```

### Рекомендуемый UI

В `Settings` добавить раскрываемый блок:

```text
📎 Прикреплённые файлы
```

Внутри таблица:

- подрядчик;
- период;
- сумма;
- статус;
- имя/путь файла;
- размер файла;
- дата изменения файла;
- кнопка скачать;
- кнопка удалить файл.

### Важное правило удаления

Удаление должно делать две вещи:

1. удалить файл с диска;
2. очистить `Payment.receipt_file = None`.

Не нужно удалять сам платёж.

### Нужные backend routes

Предложение:

```text
GET  /settings/attachments
POST /settings/attachments/{payment_id}/delete-file
```

Или встроить прямо в `/settings`, чтобы не плодить отдельную страницу.

### Защита

- Только admin.
- Обязательно CSRF после реализации.
- Нельзя принимать произвольный путь от пользователя.
- Удалять только файл, путь к которому уже лежит в `Payment.receipt_file`.
- Нормализовать путь и проверять, что он внутри `UPLOAD_DIR`.

## 5. Backup audit

### Что сейчас реально нужно бэкапить

Судя по `docker-compose.yml`, важные данные лежат на хосте:

```text
./data:/app/data
./logs:/var/log/zhkh-bot
```

Главные данные:

```text
./data/zhkh.db
./data/uploads/
.env
```

Дополнительно желательно:

```text
./logs/
docker-compose.yml
```

### Риски текущей схемы

1. SQLite-файл `zhkh.db` нельзя просто копировать в момент записи без аккуратного snapshot/backup API.
2. Upload-файлы и БД должны бэкапиться согласованно.
3. `.env` содержит секреты и Telegram token; его нужно бэкапить, но хранить защищённо.
4. В проекте есть модель `BackupHistory`, но полноценной логики backup пока нет.
5. Нет проверки восстановления.

### Минимальный backup MVP

Сделать скрипт:

```text
scripts/backup.sh
```

Что делает:

1. создаёт директорию `backups/YYYY-MM-DD_HH-MM-SS`;
2. делает SQLite online backup:

```bash
sqlite3 ./data/zhkh.db ".backup './backups/<ts>/zhkh.db'"
```

3. копирует uploads:

```bash
rsync -a ./data/uploads/ ./backups/<ts>/uploads/
```

4. копирует `.env` в backup с правами `600`;
5. пишет manifest:

```text
backup_manifest.json
```

6. упаковывает всё в tar.gz;
7. удаляет старые backups по retention, например оставить последние 14.

### Restore MVP

Сделать документ:

```text
docs/RESTORE.md
```

Минимальные шаги восстановления:

```bash
docker compose down
cp backup/zhkh.db ./data/zhkh.db
rsync -a backup/uploads/ ./data/uploads/
cp backup/.env ./.env
docker compose up -d --build
```

### Web UI для backup

В настройках добавить раскрываемый блок:

```text
💾 Резервное копирование
```

Минимальные кнопки:

- `Создать backup сейчас`;
- `Скачать последний backup`;
- `Проверить backup`;
- таблица истории backup.

### Дальше, после MVP

- автоматический backup по расписанию через APScheduler;
- запись результата в `BackupHistory`;
- backup на внешний путь, например `/mnt/backup/zhkh`;
- опционально Synology/SFTP;
- уведомление в Telegram при успешном/провальном backup;
- регулярный restore-test.

## 7. Важное предупреждение для следующего агента

Не начинать `PaymentTransaction ledger`, пока не будет проверен текущий сценарий variable-платежей без суммы. Ledger затронет модель данных и потребует миграций.

Ближайшая безопасная задача: `Прикреплённые файлы` или backup MVP. Оба не требуют менять существующую модель платежей, если делать аккуратно.
