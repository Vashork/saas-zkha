# Architecture review and dashboard fix notes

Дата ревизии: 2026-06-24
Ветка: `audit-dashboard-fixes`

## 1. Что это за проект

`saas-zkha` — локальное приложение для учета платежей ЖКХ:

- `web`: FastAPI + Jinja2 веб-интерфейс;
- `bot`: Telegram-бот на aiogram;
- `nginx`: reverse proxy;
- база: SQLite через SQLAlchemy async;
- файлы чеков: `data/uploads`;
- запуск: Docker Compose.

## 2. Исправления дашборда в этой ветке

### 2.1. Кривой список месяцев

Проблема была в `app/web/routes/dashboard.py`: список месяцев строился через вычитание 23 месяцев от текущего месяца, но год корректировался только один раз.

Пример: если текущий месяц июнь, то `июнь - 23 = -17`; код делал `+12`, получал `-5`, и такой месяц попадал в список. Это давало пустые/кривые названия месяцев и ощущение, что нормально выбирается только часть дат.

Исправлено:

- добавлен `_shift_month(year, month, offset)`;
- список месяцев теперь нормализуется через общий индекс месяца;
- в селектор попадают:
  - месяцы из существующих платежей;
  - диапазон от 24 месяцев назад до 12 месяцев вперед;
  - выбранный месяц, даже если в нем пока нет платежей.

### 2.2. Выбранный месяц не подсвечивался

Проблема: в шаблоне `dashboard.html` проверялось условие `m == month`, но переменная `month` не передавалась в контекст шаблона.

Исправлено:

- в контекст добавлен `"month": month`.

### 2.3. Не отображались долг и неоплаченные счета

Проблема: дашборд считал `total`, `pending_amount` и `overdue_amount` только по `Payment.amount`.

Но для переменных платежей `Payment.amount` может быть `NULL`, а после фиксации через Telegram заполняется только `paid_amount`. Из-за этого суммы могли отображаться как `0`.

Исправлено:

- добавлены helper-функции:
  - `_planned_amount(payment)`;
  - `_paid_amount(payment)`;
  - `_remaining_amount(payment)`;
- `total` теперь берет лучшую известную сумму:
  1. `payment.amount`;
  2. если нет — `payment.paid_amount`;
  3. если нет — `contractor.fixed_amount` для fixed-подрядчиков;
  4. иначе `0`;
- добавлены `unpaid_amount` и `unpaid_count`;
- таблица дашборда теперь показывает начислено и остаток.

## 3. Узкие места архитектуры

### P0 — критично

1. Авторизация держится на доверии к cookies.

   В cookies лежат `user_id`, `username`, `user_role`, `page_permissions`. Проверки админских прав и доступа к страницам идут по этим cookies. Пользователь может подменить `user_role=admin` и получить права администратора.

   Рекомендация:

   - хранить в cookie только signed session id;
   - на каждом запросе доставать пользователя из БД;
   - проверять `role`, `page_permissions`, `is_active` только по БД;
   - использовать `itsdangerous`, `starlette-session`, JWT с подписью или серверные сессии.

2. Деактивированный пользователь может войти.

   При логине проверяется только логин/пароль, но не `User.is_active`.

   Рекомендация:

   - в `/login` добавить условие `User.is_active == True`;
   - при деактивации пользователя инвалидировать его сессии.

3. Контейнер Telegram-бота запускается неправильно.

   В `docker/Dockerfile.bot` указано `python -m app.bot`, но в пакете `app/bot` нет `__main__.py`. Реальная точка входа — `app/bot/main.py`.

   Рекомендация:

   - заменить запуск на `python -m app.bot.main`; либо
   - добавить `app/bot/__main__.py`.

### P1 — важно

4. Уведомления не отправляются в Telegram.

   Scheduler находит срочные/просроченные платежи и меняет статус, но отправка Telegram уведомлений помечена как TODO.

   Рекомендация:

   - сделать отдельный notification service;
   - бот должен отправлять сообщения `TELEGRAM_ADMIN_ID` или связанным пользователям;
   - хранить историю отправленных уведомлений, чтобы не спамить.

5. Дублирование логики статуса платежа.

   Статус `overdue` иногда хранится в БД, а иногда вычисляется по `due_date <= today` через `payment_color_class` или `_effective_status`.

   Рекомендация:

   - выбрать один подход:
     - либо `overdue` всегда вычисляемый статус;
     - либо scheduler обновляет БД, а UI только отображает;
   - вынести в единый сервис `payment_status.py`.

6. Нет сервисного слоя.

   Сейчас маршруты напрямую работают с SQLAlchemy, файлами, расчетами и HTML-контекстом.

   Рекомендация:

   - вынести бизнес-логику в `app/services/`:
     - `payments_service.py`;
     - `dashboard_service.py`;
     - `notification_service.py`;
     - `auth_service.py`.

7. Нет миграционной системы.

   Миграции реализованы вручную через `PRAGMA table_info` и `ALTER TABLE`.

   Рекомендация:

   - подключить Alembic;
   - вести версионирование схемы БД.

8. Нет нормальной модели сессий.

   Сейчас нет таблицы сессий, нет отзыва сессии, нет `secure`/`signed` cookie.

   Рекомендация:

   - внедрить серверные сессии;
   - поставить `secure=True` при HTTPS;
   - добавить `SameSite=Lax` или `Strict`;
   - добавить CSRF-защиту для POST-форм.

### P2 — средний приоритет

9. Шаблоны повторяют navbar на каждой странице.

   Рекомендация:

   - вынести navbar в `templates/partials/navbar.html`;
   - подключать через `{% include %}`.

10. Настройки темы пишутся в разные ключи.

   Один обработчик использует `ui_theme`, другой сохраняет `theme`.

   Рекомендация:

   - оставить один ключ: `ui_theme`;
   - мигрировать старый `theme`, если он есть.

11. Путь к чекам в history отличается от payments.

   В payments используется `/uploads/...`, а в history встречается `/static/uploads/...`.

   Рекомендация:

   - везде использовать `/uploads/{{ p.receipt_file }}`.

12. Нет тестов на dashboard/date/month logic.

   Рекомендация:

   - добавить unit-тесты для `_shift_month`, `_parse_selected_period`, `_planned_amount`, `_remaining_amount`;
   - добавить интеграционный тест рендера `/` с несколькими месяцами.

13. Валидация форм размазана по handlers.

   Pydantic schemas есть, но в маршрутах они почти не используются.

   Рекомендация:

   - использовать schemas в сервисном слое или API handlers;
   - унифицировать ошибки валидации.

## 4. Мертвый код / мертвые переменные / подозрительные места

1. `app/schemas.py` выглядит почти неиспользуемым.

   Schemas описаны, но HTML routes принимают `Form(...)` напрямую, а не Pydantic-модели. Это кандидат на удаление или переиспользование в сервисах/API.

2. `Contractor` импортировался в `dashboard.py`, но не использовался.

   В этой ветке импорт удален.

3. `RedirectResponse` импортировался в `dashboard.py`, но не использовался.

   В этой ветке импорт удален.

4. `logger` в некоторых route-файлах объявлен, но почти не используется.

   Решение: либо добавить осмысленное логирование операций, либо удалить лишние loggers.

5. `_context()` в `payments.py` используется только для ошибок и создает неполный контекст.

   Риск: при ошибке форма может отрендериться без актуального списка платежей/подрядчиков.

   Рекомендация: заменить на полноценную функцию загрузки контекста платежей.

6. `due_day = min(c.due_day, 28)` в scheduler вычисляется, но не используется.

   Внутри `generate_monthly_payments()` переменная `due_day` создается, но затем используется `c.due_day`. Это мертвая переменная и потенциальный баг.

7. `vals_curr` в `analytics.py` создается, но не используется.

   В single-month ветке создается список `vals_curr = []`, но дальше он не заполняется и не нужен.

8. `TELEGRAM_ADMIN_ID` есть в конфиге, но не используется для отправки уведомлений.

   Это заготовка под notification service.

9. `BackupHistory` есть в модели, но backup script не пишет историю в БД.

   Значит страница мониторинга бекапов пока невозможна без дополнительной логики.

10. `is_allowed_file` импортируется в bot handlers, но не применяется при скачивании документа.

   Риск: бот может сохранить неразрешенный тип файла, если Telegram mime-type не соответствует ожиданиям.

## 5. План работ для агента

### Этап 1 — стабилизация запуска

1. Исправить `docker/Dockerfile.bot` на `python -m app.bot.main`.
2. Проверить `docker compose up -d --build`.
3. Проверить:
   - `docker logs zhkh-web`;
   - `docker logs zhkh-bot`;
   - `curl http://localhost/health`.

### Этап 2 — безопасность

1. Переделать auth middleware/helper:
   - не доверять `user_role` и `page_permissions` из cookie;
   - получать пользователя из БД;
   - проверять `is_active`.
2. Добавить signed session.
3. Добавить CSRF для POST-форм.

### Этап 3 — платежная модель

1. Уточнить семантику:
   - `amount` = начислено;
   - `paid_amount` = оплачено;
   - `remaining` = вычисляемое поле.
2. При оплате через Telegram для переменного платежа записывать `amount = paid_amount`, если начисление заранее неизвестно.
3. Добавить единый helper/service для расчета остатка.

### Этап 4 — дашборд

1. Проверить исправленный селектор месяцев.
2. Проверить выбранный месяц с URL вида `/?year=2026&month=6`.
3. Проверить суммы:
   - fixed unpaid;
   - fixed paid;
   - variable unpaid;
   - variable paid through Telegram;
   - overdue.
4. Добавить тесты на helper-функции.

### Этап 5 — уведомления

1. Реализовать отправку Telegram уведомлений.
2. Добавить таблицу/модель истории уведомлений.
3. Защититься от повторной отправки одного и того же уведомления.

## 6. Definition of Done для дашборда

Дашборд считается исправленным, если:

- список месяцев не содержит пустых или отрицательных месяцев;
- выбранный месяц подсвечивается в селекторе;
- `/?year=YYYY&month=M` открывает нужный период;
- карточка `Остаток к оплате` показывает сумму неоплаченных счетов;
- карточка `Просрочено` показывает просроченную часть;
- таблица показывает неоплаченные платежи, сумму начисления и остаток;
- переменные платежи с `paid_amount` не отображаются как `0` в статистике;
- нет регрессии на текущем месяце.
