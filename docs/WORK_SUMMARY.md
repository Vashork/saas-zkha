## 2. Важное ограничение текущей модели платежей

Сейчас `Payment` — это одна строка на `contractor/year/month`.

Она умеет показать:

- начислено;
- оплачено;
- остаток;
- просрочку;
- один чек в `receipt_file`.

Но это ещё не полноценный журнал частичных оплат. Если один долг закрывается несколькими платежами и несколькими чеками, лучшее долгосрочное решение — отдельная таблица операций:

```text
PaymentPeriod
- contractor_id
- year
- month
- planned_amount
- due_date
- status

PaymentTransaction
- payment_period_id
- paid_amount
- paid_date
- receipt_file
- source: web | telegram
- comment
```

### Telegram bot

Сейчас бот фиксирует оплату по тегам текущего месяца:

```text
#оплачено #slug #сумма:3200
```

Если пользователь пришлёт чек без тегов, текущая версия ещё не задаёт уточняющие вопросы. Следующий этап — добавить сценарий диалога через FSM: бот спросит подрядчика, сумму и период, а потом сохранит оплату.

Для старого долга нужен явный период:

```text
#оплачено #slug #сумма:1000 #период:2026-06
```

## 5. Рекомендуемый порядок QA текущего пакета

1. `git pull origin audit-dashboard-fixes`.
2. `docker compose up -d --build`.
3. Открыть `Платежи`, выбрать прошлый месяц.
4. Создать платёж за прошлый месяц со статусом `Ожидает оплаты` и убедиться, что поля даты оплаты/чека не предлагаются.
5. Создать/изменить платёж со статусом `Оплачено` и убедиться, что поля даты оплаты/чека появились.
6. Изменить `Оплачено` обратно на `Ожидает` или `Просрочено` и проверить, что `Оплачено` обнуляется.
7. Проверить фильтр `Просрочено`: визуально просроченная запись должна появиться там.
8. Проверить dashboard: `Всего к оплате` может быть равно `Из них просрочено`, если весь неоплаченный долг уже просрочен.
9. Открыть `/payments?year=2026&month=5`, сбросить/потерять сессию, залогиниться и убедиться, что вернуло обратно на майские платежи.
